# OpenClaw-Tips（HJN 实战最佳实践）

> 目标：把我（HJN）实际用 OpenClaw 提升工作/生活效能的“可复用套路”沉淀成文档。
> 结构严格按：**实际产出应用｜学习｜效率管理｜个人定制｜Tips（skill / GitHub / dashboard）**。

---

## 0) 一句话总纲（先记住）

- **把碎片输入变成交付物**：灵感/语音/聊天 → 提纲 → 直接生成 `md/html/code` → 入库（GitHub/Airtable）→ 可复盘可迭代。

---

## 1) 实际产出应用：Vibe Project

### 痛点
- 灵感经常出现在手机不在手/走路/地铁等场景，回到电脑很难继续挖掘
- 更难“直接生成一个可交付文件”用于分享/协作

### 方案（我怎么用 OpenClaw）
- 先把一个项目压成**小提纲**（3-7 条即可）
- 让 OpenClaw 直接生成：
  - `README.md` / 方案 `md`
  - 可预览分享的 `html` one-pager
  - 或者把 chatbox 的脑暴 case **直接落到仓库**（notes/ + assets/）

### 效果
- 真正做到**手机端 vibe coding / vibe working**：抓住即时灵感 → 立即产出可分享交付物
- 产物有“文件形状”，方便自己复用，也方便丢给别人继续做

### 优化实践案例（我们已经做过）
- **Love-Spinner（恋爱转盘）**：
  - 你提出交互：“问题在中间，左右男女选 ABCD，聊完再选一个协商答案”
  - 我补齐交付物：题库 `question_bank_v0.json` + 交互规范 `INTERACTION_SPEC.md`
  - 并推到 GitHub：`HJN-Bot/Love-Spinner`（别人可直接开工）

## 2) 学习

### 痛点
1) 收集材料费时间
2) 学习输出无人 review（缺反馈回路）
3) 学习项目难以保存成“个人知识库”（过几天就找不到）

### 方案
- 工具不重要：GitHub / Notion / Obsidian 都行
- 让 OpenClaw 做两件事：
  1) **准备材料**（链接/摘要/参考实现/练习题）
  2) **把输出变成可 review 的产物**（代码/笔记/小练习，提交到仓库）

### 效果
- 学习输出能被 review、能被迭代，最后沉淀为可检索的知识库

### 优化实践案例
- **春节学习清单（Phase One/Two）**：把大而全学习拆成 3-5 天 checklist（只学能验收的 20%：E2E复现/Review+patch/rerun/Evals+AB/LLM验收）。仓库：`HJN-Bot/phase-one-phase-two-cny-checklist`。
- **Prompt+Skills 资源沉淀**：把零散的“好用网站/技巧”放进看板 threads，形成“随用随取”的学习入口，避免重复搜索。

## 3) 效率管理

### 痛点
- 学习/工作信息纷杂：记录、输入、输出缺系统性管理
- 多平台切换成本高，容易漏

### 方案（平台化整合）
- **底层：Airtable** 作为后方大数据库（Tasks / Ideas / Goals / Weekly / 金句 / 顿悟时刻等 sheet）
- **中层：Dashboard** 展示近 2 周 Top8（低噪音聚焦此刻最重要的输入与输出）
- **表层：交互约定**：当我梳理进度/复盘时，必须把结论回流到 Airtable，并刷新 dashboard

### 效果
- 输入记录更全面、同步更快、追踪更可视化

### 优化实践案例
- **自动同步 + 下班地铁提醒**：定时从 Airtable 拉取近 14 天 Top8 → 刷新 `dashboard/index.html` → 在 Discord 发“标题+相关+状态+一句话”，并提醒是否要 Idea→Task/清理长期未动。
- **语音想法直接变 Task**：你口述“明天要整理 cloud 交互体验并生成网页分享”→ 我直接创建 Airtable 任务（High），并立刻刷新 dashboard，保证第二天开工不丢。

## 4) 个人定制

### 痛点
- 需要一个对我个人上下文“足够熟悉”的长期角色
- 不只是做任务：要能按我喜欢的方式梳理情绪/情感，并把结论回流到系统

### 方案
- **个人信息底座**：能力画像/顾问团指导/AI 技术自评/读书清单等（作为长期上下文）
- **交互风格定制**：专业高效、先结论后展开（你明确偏好）
- **回流机制**：关键结论→写入 README/notes + Airtable（可检索、可复盘）

### 效果
- “够懂我、够有解决力、能回流到系统”——不仅是聊天爽，而是能形成长期积累

### 优化实践案例
- **内容分享任务**：你提出要整理“cloud/OpenClaw 的交互体验与使用场景”，并且要讨论“与 AI 的边界” → 直接沉淀成 Airtable 高优先级任务，并要求产出网页（可分享、可迭代）。

## 5) Tips（好用的具体建议）

### 5.1 GitHub（账号与习惯）
- 推荐：给 OpenClaw 配置一个专用 GitHub 账号/组织（你这里用的是 `HJN-Bot`）
- 所有交付物尽量落到 repo：
  - `README.md`：怎么用 + 交付物 + 下一步
  - `notes/`：脑暴与会议纪要
  - `assets/`：图片、海报、截图

### 5.2 常用 skill 推荐（按你提到的方向补齐）
- **前端设计类**：`frontend-design`
  - 用法：生成 one-pager / demo 页面 / 小应用 UI（避免“AI土味页面”）
- **图表/可视化整合类（建议补）**：
  - 现在常见做法：用 `coding-agent` 直接生成可运行的 HTML 报表（ECharts/Plotly），或把数据整理成 Markdown 报告
  - 如果你希望更“产品化”：后续可以专门做一个“图表报告 skill/脚本”放进本仓库
- **情感交互类（建议补）**：
  - 方向：情绪梳理→需求识别→可执行对话脚本→回流到个人画像/规则
  - 现成可用套路：用《爱的艺术》框架把“吸引下降”拆成：热度回落 vs 连接质量下降，并落到可执行三问
- **工程落地类**：`github` / `coding-agent`
  - 用法：批量生成文件、写 PRD/Spec、跑评测口径、落地 repo
- **信息获取类**：`summarize` / `blogwatcher` / `bird`
  - 用法：材料准备、行业情报、线程总结

### 5.3 Dashboard（你这套系统的“中控台”）
- 访问：`http://54.169.36.2:8080/`（Basic Auth）
- 服务脚本：`/home/ubuntu/.openclaw/workspace/tools/dashboard_web.mjs`
- 页面文件：`/home/ubuntu/.openclaw/workspace/dashboard/index.html`

**推荐你怎么用：**
1) 只看 Top8（近 14 天），强制低噪音
2) 从卡片回到 Airtable 查细节
3) 一切自动化执行项目（同步/提醒/监控）都在这里可见
